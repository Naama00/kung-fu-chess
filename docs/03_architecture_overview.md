# סקירה ארכיטקטונית של המערכת (Architecture Overview)

## 1. הרכיבים המרכזיים

### א. מנוע המשחק והניהול (`GameEngine`)
* **תפקיד:** מהווה את נקודת הממשק הראשית (Facade) של הלוגיקה מול בקרי קלט ומערכות תצוגה.
* **אחריות:** 
  * מנהל את שעון המשחק הראשי (`wait`).
  * מנתב פקודות תנועה (`requestMove`) ל-`RealTimeArbiter` לאחר אימות ראשוני.
  * מנהל את תור החלפת התורות במצב שאינו סימולטני.
  * מריץ את בדיקת המהלכים מראש (`PremoveQueue`).

### ב. בורר זמן האמת (`RealTimeArbiter`)
* **תפקיד:** מנהל את ההיבטים הפיזיקליים הדינמיים של המשחק.
* **אחריות:**
  * החזקת רשימת הכלים שנמצאים כרגע בתנועה (`Motion`).
  * מעקב אחר זמני הצינון באמצעות רכיב עזר ייעודי (`CooldownTracker`).
  * החלת אינטרפולציה ועדכון מיקומי כלים בזמן התקדמות השעון.
  * ניהול פתרון התנגשויות גיאומטריות באמצעות `CollisionDetector` ו-`CollisionResolver`.

### ג. מנוע החוקים הסטטי (`RuleEngine`)
* **תפקיד:** משמש כפונקציה טהורה (Pure Logic) לבדיקת חוקיות מהלכים.
* **אחריות:**
  * בודק האם מהלך מנקודה א' לנקודה ב' חוקי מבחינה גיאומטרית עבור כלי ספציפי, בהתבסס על מצב הלוח הנוכחי.
  * מאציל את הבדיקות לחוקים ייעודיים לכל כלי (`PieceRules`) תוך שימוש ב-Factory Pattern.
  * אינו יודע דבר על זמני צינון או תנועה פיזית – אחריותו מוגבלת לחוקי שחמט סטטיים בלבד.

### ד. תור מהלכים מראש (`PremoveQueue`)
* **תפקיד:** אוגר פקודות של השחקן שנשלחו עבור כלים "עסוקים".
* **אחריות:** 
  * שומר על יעד התנועה ומבצע בדיקה חוזרת ברגע שהכלי מתפנה.
  * מאפשר החלפה חלקה של כלים שקודמו (Promoted) בתוך התור מבלי לשבור את רצף הפקודות.

## 2. זרימת בקשת תנועה (Move Request Flow)
1. ה-`Controller` (או מערכת הקלט הגרפית בעתיד) קורא ל-`requestMove(from, to)`.
2. ה-`GameEngine` בודק האם הכלי המבוקש עסוק:
   * **אם הכלי עסוק:** הבקשה נשלחת לרישום ב-`PremoveQueue` ומחזירה תגובה מתאימה.
   * **אם הכלי פנוי:** המנוע מעביר את המהלך לבדיקה ב-`RuleEngine`.
3. אם המהלך חוקי, ה-`RealTimeArbiter` מתחיל את תהליך התנועה הפיזיקלית:
   * יוצר אובייקט `Motion` חדש ומעדכן את מצב הכלי ל-`Moving`/`Airborne`.
   * במידת הצורך, הלוח הלוגי מעודכן מיידית.
